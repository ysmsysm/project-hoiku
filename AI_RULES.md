# AI_RULES.md

# Project Hoiku AI Development Rules

このファイルはAI（Codex）が実装時に必ず従うルールです。

---

# 最優先事項

Project Hoikuの目的は

「保育園準備を30秒で終わらせる体験」

を実現することです。

見た目を新しくすることではありません。

UIの一貫性を最優先してください。

---

# 実装方針

以下を優先してください。

1. 保守性
2. 可読性
3. 型安全
4. コンポーネント再利用
5. 将来の拡張性

---

# UI変更禁止

ユーザーの承認なしに以下を変更してはいけません。

・画面構成
・カードデザイン
・余白
・配色
・アイコン配置
・ボタン配置
・ナビゲーション
・操作方法

内部実装だけ変更してください。

---

# ロッカーUI

確認画面は

「ロッカーを見ながら丸を付ける」

UIを採用しています。

このUIはProject Hoikuの特徴です。

以下は禁止です。

・＋−入力
・Stepper
・数字入力フォーム
・スライダー
・リスト型入力への変更

---

# コンポーネント

既存コンポーネントを優先して利用してください。

必要以上に新規コンポーネントを増やさないでください。

---

# State管理

状態管理は一箇所に集約してください。

状態を複数画面で重複管理しないでください。

---

# Storage

localStorageへ直接アクセスせず

lib/storage.ts

経由で利用してください。

将来的に

Supabase

Firebase

へ差し替えられる構造にしてください。

---

# 型

型は

types/

へ配置してください。

anyは禁止です。

型安全を優先してください。

---

# Build

実装終了前に

npm run build

を実行してください。

TypeScriptエラーがある状態で終了しないでください。

---

# 実装完了時

毎回最後に

## 変更内容

## 追加ファイル

## 更新ファイル

## 今後の改善候補

を必ず報告してください。

---

# 勝手にやってはいけないこと

・UI刷新
・画面構成変更
・命名変更
・ファイル大量移動
・不要なリファクタリング
・ライブラリ追加
・デザイン変更

---

---

# コード品質

コードは

シンプル

読みやすい

保守しやすい

を優先してください。

短いコードより理解しやすいコードを採用してください。

---

# Project Hoikuの思想

Project Hoikuは

「便利なアプリ」

ではなく

「毎日30秒を積み重ねて家族の負担を減らすプロダクト」

です。

すべての実装はこの思想に従ってください。---

# UI変更承認ルール（最重要）

UI・UXに影響する変更は、ユーザーの承認を得るまで実装しないでください。

以下に該当する変更は必ず事前確認してください。

・画面レイアウト
・入力方法
・導線
・タブ構成
・カード構成
・ボタン配置
・色
・余白
・アニメーション

内部実装（型、状態管理、リファクタリング、保守性向上）は事前承認不要です。
